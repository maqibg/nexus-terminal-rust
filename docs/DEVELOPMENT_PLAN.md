# DEVELOPMENT PLAN

> 文档类型：完整开发计划（执行版）  
> 项目：`nexus-terminal-rust`  
> 目标形态：`Tauri 2 + Rust`（UI 不变）  
> 更新时间：2026-02-11

---

## 1. 计划总览

### 1.1 总目标

在保证现有显示效果与核心业务能力不退化的前提下，完成从 Electron+Node 到 Tauri2+Rust 的生产级迁移，并达到以下指标：

- 冷启动 <= 2.5s
- 热启动 <= 1.0s
- 空闲内存下降 >= 40%
- 核心功能回归通过率 100%

### 1.2 计划方式

- 采用 **阶段式并行可回退** 策略。
- 每阶段都有“进入条件、产出物、退出条件”。
- 每周固定评审里程碑，避免延迟到末期集中爆发。

## 6. 质量门禁与阶段放行标准

### 6.1 每阶段统一门禁

- 编译通过
- 自动化测试通过
- 文档同步更新
- 无未记录高风险改动

### 6.2 关键门禁指标

- Rust 核心覆盖率 >= 80%
- P0/P1 契约测试 100%
- 核心 E2E 全绿
- 启动与内存指标不回退

## 7. 风险管理计划

## 7.1 主要风险矩阵

1. **实时链路兼容风险（高）**
   - 对策：协议回放测试 + 双端日志对比
2. **迁移周期膨胀（中高）**
   - 对策：按域切片、每两周里程碑审查
3. **Windows 权限与签名问题（中）**
   - 对策：提前验证 capability 与签名链
4. **数据迁移失败（高）**
   - 对策：迁移前备份 + 自动回滚 + 演练

### 7.2 风险处理机制

- 每周固定风险审查
- 风险分级：P0/P1/P2
- 每个高风险必须有 owner 与截止时间

---

## 8. 需求变更控制

### 8.1 变更分级

- 轻微变更：不影响里程碑
- 中等变更：影响单阶段
- 重大变更：影响总周期或架构

### 8.2 审批机制

- 重大变更必须经技术负责人 + 产品负责人双签
- 变更通过后 24h 内更新计划与文档
